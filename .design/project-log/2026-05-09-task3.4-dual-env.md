# Project Log: Phase 3 Dual Environment Variable Export

## Task Overview
Implemented Phase 3 of the rename strategy, focusing on dual environment variable export. This ensures that both the legacy `GROVE` and new `PROJECT` prefixed environment variables are available to agents and tools, facilitating a smooth transition.

## Changes

### Hub Dispatcher
- Updated `pkg/hub/httpdispatcher.go` to include `FABRIC_PROJECT_ID` alongside `FABRIC_GROVE_ID` in the `resolvedEnv` map sent to brokers.

### Runtime Broker
- Updated `pkg/runtimebroker/start_context.go` to inject both `FABRIC_PROJECT_ID`/`FABRIC_GROVE_ID` and `FABRIC_PROJECT_PATH`/`FABRIC_GROVE_PATH` into the agent container environment.
- Updated `pkg/runtimebroker/hubenv.go` to include the new environment variables in `safeEnvLogKeys` for safe logging.

### Agent Runtime
- Updated `pkg/agent/run.go` to:
    - Set `FABRIC_PROJECT` alongside `FABRIC_GROVE`.
    - Fall back to `FABRIC_PROJECT_ID` if `FABRIC_GROVE_ID` is missing when resolving project ID for labels.
    - Use both environment variables for the `fabric.grove_id` container label.

### Fabrictool & Telemetry
- Updated `pkg/fabrictool/hooks/handlers/telemetry.go` to check `FABRIC_PROJECT_ID` for the `grove_id` metric attribute.
- Updated `pkg/fabrictool/telemetry/gcp_exporter.go` and `pkg/fabrictool/telemetry/providers.go` to include `FABRIC_PROJECT_ID` when building resource attributes and log labels.

### CLI & Sync
- Updated `pkg/config/project_marker.go`: `IsHubContext()` now also checks for `FABRIC_PROJECT_ID`.
- Updated `pkg/hubsync/sync.go`: `EnsureHubReady()` now correctly resolves the project ID from either `FABRIC_GROVE_ID` or `FABRIC_PROJECT_ID` when running in a hub-connected container.

### Gemini Harness
- Updated `pkg/harness/gemini/embeds/settings.json` to allow the new project-prefixed environment variables.

## Verification
- Ran `make build` to ensure no compilation errors.
- Verified that all key locations identified in the task were updated.
- Proactively identified and updated additional locations in `fabrictool` and `config` to ensure full compatibility.

## Observations
The transition to dual environment variables provides immediate backward compatibility for existing harnesses and tools while enabling new ones to use the `PROJECT` prefix.
